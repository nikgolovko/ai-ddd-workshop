[.lead]
This document provides two things for Step 6 of the VQS DDD Workshop:

. A *blank template* for facilitating a User Story Mapping session (Section 1).
. A *fully worked example* covering the complete VQS workflow across three releases (Section 2).

Duration: *90 minutes*. +
Goal: Align development priorities with real user workflow. Define the MVP slice. Make scope trade-offs explicit and visible. Ensure EU AI Act audit trail stories appear in the MVP, not a later release.

=== How a User Story Map Works

[cols="2,8", options="header"]
|===
| Dimension | Content

| Horizontal (left → right)
| The *workflow backbone* — the user activities in the order they happen in real life. Maximum 6–8 backbone steps. These are verbs: Detect, Assign, Investigate, Override, Escalate, Retrain.

| Vertical (top → bottom)
| *Story depth* — the stories that flesh out each backbone step. Stories nearer the top are more important. Draw a horizontal release line: everything above is MVP, everything below goes to a later release.

| Rows (user roles)
| Optional. Place user role cards above the backbone to show which actor drives each step.
|===

=== User Story Format

Use the standard Given/When/Then or role/feature/benefit form.

Recommended short form for story cards:

----
As a [role]
I need to [action]
So that [outcome]
----

=== Release Line Definition

[cols="2,8", options="header"]
|===
| Release | Criterion for inclusion

| MVP
| Required to deliver the core value proposition AND meet all regulatory compliance obligations. Nothing more.

| Release 2
| Makes the system smarter or more efficient. Delivers the Learning Loop. Not required for go-live.

| Release 3 / Backlog
| Future value. Depends on Release 2 outcomes or organisational readiness. Do not over-specify.
|===

== Filled Example — VQS Edge Inspection System

=== Session Metadata

[cols="2,5", options="header"]
|===
| Field | Value

| Date
| 2026-04-08

| Facilitator
| Dr. N. Golovko

| Participants
| Tim B. (Product Owner) · Petra K. (Quality Engineer) · Sven M. (ML Engineer) · Jana R. (Plant Manager) · Dev Lead

| Input from Step 5
| Context Map with 6 boundaries · ACL contract stubs · 4 EU AI Act boundaries · open question on DriftSignal environmental context
|===

=== User Roles

[cols="3,7", options="header"]
|===
| Role | Description / goal in the system

| Quality Engineer (Petra)
| Operates on the shop floor. Needs fast, confident decisions. Wants to understand when the model is wrong without reading a dashboard.

| ML Engineer (Sven)
| Responsible for model quality. Needs early drift signals and structured feedback to improve models without waiting for weekly reports.

| Plant Manager (Jana)
| Accountable for line throughput and defect escape rate. Needs visibility across shifts without being in the loop for every decision.

| Case Manager
| Reviews RCA Recommendations and approves or rejects them before any corrective action. Needs enough context to make an informed decision quickly.

| Regulator / Auditor
| Needs a complete, tamper-proof audit trail of every AI-assisted decision and every human override for EU AI Act compliance.
|===

=== Workflow Backbone

[cols="1,3,7", options="header"]
|===
| # | Backbone step | User activity description

| 1
| Detect
| The system detects a potential defect on a production unit in real time. The edge model runs inference and produces a Decision.

| 2
| Assign
| A Decision (SCRAP / REWORK / PASS) is assigned to the InspectionCase from the active DecisionBand. The panel is physically routed by the diverter.

| 3
| Review
| The Quality Engineer is alerted and reviews the HMI. She confirms or overrides the Decision, selecting a reason code if overriding.

| 4
| Investigate
| When override patterns breach the threshold, a root cause analysis is triggered. An LLM analyses the defect pattern and produces a hypothesis, translated into a Recommendation by the ACL.

| 5
| Approve
| The Case Manager reviews the Recommendation and approves or rejects it. This is the human oversight gate required by EU AI Act.

| 6
| Retrain
| When drift is confirmed, a retraining pipeline is initiated with Plant Manager approval. A new model version is produced, validated, and deployed to the edge fleet.
|===

[cols="1,6,2", options="header"]
|===
| ID | Story | Backbone step

| EU-01

=== Scope Decisions Made

[cols="5,5", options="header"]
|===
| Item moved out of MVP | Reason

| Automated retraining pipeline (R3-01, R3-02)
| No actor needs automated retraining to achieve the 30% escape rate reduction goal in 6 months. Manual-trigger retraining with Plant Manager approval is sufficient for MVP.

| Plant Manager shift dashboard (R3-04)
| The Plant Manager's primary need is addressed by the AuditLog export. A dedicated dashboard adds UX value but does not reduce defect escape rate directly.

| Environmental context logging (R3-03)
| Sensor infrastructure for lighting and temperature logging is not available at MVP. Required for DriftSignal quality improvement — schedule for Release 3.

| Synthetic data generation (R3-05)
| DataGap detection is sufficient for MVP. Automated SyntheticSample generation requires GenAI pipeline work that is out of scope for the initial release.

| Override rate trend visualisation (R2-05)
| ML Engineer can monitor override rates via raw AuditLog export in MVP. A visualisation layer is a Release 2 quality-of-life improvement.
|===

[TIP]
====
*Facilitator note for Step 7:* The Impact Map must connect every MVP story to one of the four actors and their measurable impact. EU stories (EU-01 through EU-05) map to the Regulator actor. Non-functional stories (NF-01 through NF-04) typically map to the Quality Engineer or Plant Manager actors. Any deliverable in the map that cannot be traced to an actor impact should be moved to the cut list.
